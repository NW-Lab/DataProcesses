# ADR 0003: 外部アドイン機構は将来の拡張方針として保持し、実装は保留する

- **状態:** Accepted — implementation deferred
- **日付:** 2026-07-18

## 背景

DataProcessesでは、標準Blockを継続的に追加する一方で、研究室、機器ベンダー、特定のデータ形式、実験的な信号処理など、用途固有のBlockを本体リポジトリへ集約し続けることは望ましくありません。将来的には、外部開発者が別プロジェクトでBlockを実装し、DataProcessesへ追加できる拡張モデルが必要です。

現在の`DataProcesses.Plugin.Abstractions`には、Block実行のための`INode`、生成のための`INodeFactory`、提供元カタログのための`INodePlugin`が定義されています。ただし、外部アセンブリの発見、マニフェスト検証、読み込み、依存関係の分離、互換性確認、信頼判断、更新管理を行うアドインホストはまだ実装していません。

## 決定

製品、利用者向け文書、画面、コミュニティ運営では、外部から追加される拡張を**アドイン（Add-in）**と呼びます。既存の`INodePlugin`などのコード上の識別子は、現段階では不用意に改名しません。これは、現在の公開契約を安定させ、将来の互換性判断を明確にするためです。

外部アドインを将来の正式な拡張モデルとして目指します。ただし、MVPおよび現在のプレアルファ段階では、外部アドインの検出・読み込み・配布機構を実装しません。組み込みBlockは`DataProcesses.Nodes.BuiltIn`で管理し、標準Blockの追加は本体へのPull Requestで受け入れます。

| Blockの性質 | 想定する受け入れ先 |
|---|---|
| 多くの利用者に共通し、長期的な保守主体が明確な標準Block | DataProcesses本体へのPull Request |
| 特定機器、研究用途、組織固有形式、実験的処理、任意の外部依存を持つBlock | 将来の外部アドイン |
| 拡張SDK、アドインローダー、アドイン検証、テンプレート | DataProcesses本体 |

将来の外部アドインは、`DataProcesses.Plugin.Abstractions`のみを参照してBlockを提供することを原則とします。アドイン実装からDesktop、Coreの内部実装、シリアライザー固有型、DIコンテナー固有型への直接依存は許可しません。

## 現時点で実装しないもの

次の機能は、設計・運用方針が固まるまで実装しません。

| 保留機能 | 保留理由 |
|---|---|
| 外部DLLの自動検出・読み込み | 配置場所、依存関係、失敗時の回復、ライフサイクルを決める必要があるため |
| `addin.json`または`plugin.json`の正式スキーマ | 互換性、署名、配布形式、作者情報の必要項目を決める必要があるため |
| アドイン一覧・有効化・無効化画面 | UX、永続化、再起動要件、障害表示の方針を決める必要があるため |
| アドインレジストリ、マーケットプレイス、自動更新 | 信頼、審査、ライセンス、更新責任の運用を決める必要があるため |
| 未信頼アドインの隔離実行 | Worker方式、権限、IPC、障害分離、性能目標を決める必要があるため |

この決定は、任意の外部DLLを同一プロセスへ読み込めることを約束するものではありません。アドインがデータ取得やPythonなどの外部プロセスを伴う場合は、将来のWorkerアドインモデルとして別プロセス実行を検討します。

## 再検討時の最低要件

アドイン実装を開始する前に、次の項目をADRおよび仕様書で確定します。

1. ホストと`DataProcesses.Plugin.Abstractions`のバージョン互換性方針。
2. アドインID、Block型ID、発行者識別子、マニフェストの正式スキーマ。
3. 配布単位、配置先、依存関係の解決、インストール・削除・更新の手順。
4. 信頼済みアドインの定義、署名または許可リストの要否、読み込み失敗時の表示と無効化手順。
5. 同一プロセスのManaged Add-inと別プロセスのWorker Add-inの選択基準。
6. SDKテンプレート、最小サンプル、契約テスト、互換性検証の提供方法。
7. コミュニティ提供アドインの一覧・レビュー・保守責任に関する運用方針。

## 結果

本体は、汎用的で保守可能な標準Blockに集中できます。外部開発者に対しては、将来アドインを独立配布できる拡張方向を明示しつつ、プレアルファ段階で未確定なランタイム、互換性、信頼、運用の約束をしません。

このADRは、外部アドイン機構をいつ実装するかを決めるものではありません。実装の着手は、実行・利用の経験を踏まえ、上記の最低要件について合意できた時点で再評価します。
